## Why another design system?

Every team that reaches a certain scale eventually asks the same question: should we adopt an existing component library, or build our own? The answer is almost always "adopt" — and for good reason. Existing libraries come with thousands of hours of a11y work, browser quirks fixes, and API polish already baked in.

We asked the same question. Then we looked at what we actually needed: strict design-token boundaries, a CSS-first architecture with no Tailwind dependency, full TypeScript strictness, and a component API that doesn't abstract away the DOM. None of the options we evaluated hit all four constraints without heavy patching. So origam was born.

## CSS-first, JS as a last resort

The guiding principle behind every component is: **express it in CSS before reaching for JavaScript**. If a layout can be done with `grid-template-areas`, it does not need a `ResizeObserver`. If a color blend can be achieved with `color-mix(in srgb, …)`, it does not need a JS color-math library. If an accordion can be built with `<details>/<summary>`, it does not need a custom state machine.

This principle keeps bundles small, animations declarative, and theming free — CSS variables propagate into every component automatically. When a browser truly cannot handle a CSS feature, we branch via a single `useCssSupport()` composable so the detection logic stays auditable in one place instead of scattered across components.

## Design tokens as the single source of truth

Color, spacing, typography, radius, elevation — all of it lives in `tokens/` as Tokens Studio JSON (DTCG format), processed through Style Dictionary v4. There are three tiers:

- **Primitive** — raw values (`color.neutral.500`, `space.4`).
- **Semantic** — intent (`color.surface.default`, `color.action.primary.bg`).
- **Component** — per-component references (`btn.background-color`).

Components consume semantic or component-tier variables only. No raw hex anywhere. Switching themes is a single `data-theme` attribute on `<html>` — every component updates because every component only knows about variables, not values.

## Accessibility is not an afterthought

WCAG 2.2 AA compliance is a requirement, not a nice-to-have. Every interactive component ships with keyboard navigation, visible focus indicators, correct ARIA semantics, and an axe-core test. The a11y suite runs in CI — a regression blocks the merge.

We follow the W3C's first ARIA rule: **no ARIA is better than bad ARIA**. When a native HTML element covers the semantic need, we use it. `<button>` beats `<div role="button">` every time. `<details>/<summary>` beats a custom accordion. `<time datetime="…">` beats a plain span with a date string.

## Eat your own dog food

This site — origam.dev — is built entirely with origam. The navigation, cards, buttons, chips, and code blocks you see are the same components shipped in the npm package. If a component has a rough edge here, it gets fixed in the library. The site is the most honest benchmark we have.

## No bullshit defaults

origam does not install Tailwind, does not pull in a CSS reset you didn't ask for, and does not register 95 components globally by default. You import what you use. Tree-shaking works because every component is a named export. The tokens build is a separate step so you can use the token system without the component layer if that is all you need.

We are building this in public. Issues, PRs, and opinions are welcome on GitHub.

```bash
npm install origam
```
